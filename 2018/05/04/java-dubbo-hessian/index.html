Dubbo协议与Hessian Lite序列化

2018-05-04
Dubbo

dubbo是一个专注于服务治理的高性能rpc框架，支持不同服务之间的远程调用时必然的，dubbo协议与序列化的存在是不同服务沟通的桥梁，以下是本人对dubbo协议与hessian序列化的一点研究心得，如有错误请指出。

存在的意义

dubbo协议作为其rpc通信的一种约定或者规则，rpc通信的双方必须遵守这一约定，来区分是请求、响应还是事件，同时还能表达一些行为特征，比如是单向的还是双向的等等，协议的存在便是定义规格。
协议只是定义了整体的通信信息框架，但是在具体的细节方面还需要一种方式来保证，也就是序列化，我的理解如下：

  • 语义定义：类似人与人之间进行交流，想把自己的意思传达给对方，需要一种双方都懂的语言，序列化的存在便是定义了一组语义，发送方利用特定的编码方式进行编码，接收方利用特定的解码方式可以得到正确的信息。
  • 安全性：序列化后的数据为一组二进制流，而非具体的字符串，没有正确的解码方式是无法获得正确信息的。
  • 性能：通信双方的通信方式是字节流而非字符流，性能显然要好的多。

dubbo协议

以下来分别解释各个字段的含义：

  • Magic High: dubbo协议的魔数高8位，固定为0xda,何为魔数，类似java类字节码的魔数，是一个标识符，说明这是dubbo协议。
  • Magic Low: dubbo协议的魔数低8位，固定为0xbb
  • Serialization id: 序列化类型，占用4位，dubbo支持类型：
    2 hessian
    3 java
    4 compacted java
    6 fastjson
    7 native java
    8 kryo
    9 fst
  • event: 表示该消息是否为事件，比如心跳事件，为1时候表示事件，占用1位
  • Two way: 只在请求时有用，表示是否需要服务器返回response，为1时表示需要，占用1位
  • Req/res:标识是请求还是响应，request: 1, response: 0，占用1位
  • status: 仅对于response有意义，用于表示response的状态，占用1字节，具体含义如下：
    20 OK
    30 客户端超时
    31 服务端超时
    40 错误的请求
    50 错误的响应
    60 找不到服务
    70 服务出错
    80 服务器出错
    90 客户端错误
    100 线程池打满
  • request id: 标识一个唯一的请求, 占用8个字节，响应此请求时也要利用相同的id
  • data length:这个字段占用4个字节，意思很明显是数据体的长度，单位是字节

native java: 原生java序列化实现
compacted Java：压缩java序列化，主要是在原生java序列化基础上，实现了自定义的类描述符写入和读取，写Object类型的类描述符只写入类名称，而不是类的完整信息。这样有很多Object类型的情况下可以减少序列化后的size。
java: 只是对原生java序列化和压缩java序列化的封装。
kryo,fst只是专门针对java语言的，相对与要比hessian性能高点（未实际测试过），但是hessian支持跨语言

hessian lite序列化

hessian lite并非是原生的hessian2序列化方式，而是经过阿里巴巴修改过的，是dubbo默认启用的序列化方式。
